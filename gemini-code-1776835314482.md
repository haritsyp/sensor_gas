# Product Requirements Document (PRD)
**Proyek:** Sistem Monitoring Sensor Gas (Modbus TCP Client & API Forwarder)  
**Versi:** 1.1.0  
**Status:** Draft / Pengembangan  
**Bahasa Pemrograman:** Golang

---

## 1. Pendahuluan
### 1.1 Latar Belakang
Sistem ini dirancang untuk melakukan pembacaan data secara real-time dari perangkat sensor gas yang terhubung melalui PLC menggunakan protokol Modbus TCP/IP. Selain membaca data, sistem ini juga bertugas meneruskan (forward) hasil pembacaan tersebut ke server/sistem lain melalui pemanggilan API HTTP GET.

### 1.2 Tujuan
Membangun aplikasi backend ringan yang mampu menarik data dari register PLC, mengonversinya menjadi nilai konsentrasi gas, dan secara otomatis mengirimkan nilai tersebut ke API endpoint yang telah ditentukan secara real-time.

---

## 2. Spesifikasi Teknis
### 2.1 Konfigurasi Modbus (Sumber Data)
- **Protokol:** Modbus TCP/IP
- **IP Address PLC:** `192.168.31.109`
- **Port:** `502`
- **Slave ID (Unit ID):** `247`
- **Tipe Register:** Holding Registers
- **Metode Pembacaan:** Polling (setiap 1 detik)

### 2.2 Konfigurasi API (Tujuan Data)
- **Metode:** HTTP GET
- **Base URL Endpoint:** `http://endpoint/api/set_sensor`
- **Parameter URL:** `device_id` (ID Sensor) dan `value` (Hasil bacaan gas)
- **Concurrency:** Request HTTP harus berjalan asinkron (menggunakan Goroutine) agar tidak menahan/memblokir proses polling Modbus.

### 2.3 Stack Teknologi
- **Bahasa:** Go (Golang) 1.18+
- **Library Modbus:** `github.com/goburrow/modbus`
- **Library HTTP:** `net/http` (Standard Library Golang)

---

## 3. Fitur Utama
### 3.1 Konektivitas Modbus
- Membuka dan menjaga koneksi TCP ke PLC.
- Memiliki mekanisme *timeout* (5 detik).
- Menutup koneksi secara aman (*graceful shutdown*).

### 3.2 Data Acquisition & Parsing
- Membaca 3 register holding mulai dari alamat `0`.
- Menggabungkan High Byte dan Low Byte (Big Endian) dari tiap register menjadi tipe data `float64`.
- Melakukan penyesuaian skala/scaling (pembagian) jika diperlukan.

### 3.3 Integrasi API (Data Forwarding)
- Mengirimkan hasil pembacaan ke API setelah konversi data selesai.
- Request dikirim ke endpoint dengan format:  
  `http://endpoint/api/set_sensor?device_id={id}&value={result}`
- Menggunakan *HTTP Client Timeout* (misal: 3 detik) agar *request* yang *hang* tidak menumpuk di memori.

### 3.4 Monitoring & Logging
- Menampilkan log hasil bacaan ke terminal.
- Menampilkan log status HTTP response (misal: `200 OK` atau `500 Internal Server Error`).
- Menangani error koneksi Modbus maupun error koneksi API tanpa menghentikan aplikasi.

---

## 4. Alur Kerja (Workflow)
1. **Inisialisasi:** Aplikasi memuat konfigurasi IP Modbus, HTTP Client, dan pemetaan Device ID.
2. **Handshake:** Aplikasi melakukan koneksi ke PLC.
3. **Looping (Setiap 1 Detik):**
   - Kirim permintaan `ReadHoldingRegisters`.
   - Terima dan konversi byte array menjadi nilai konsentrasi Gas 1, Gas 2, dan Gas 3.
   - Menjalankan proses HTTP GET secara *concurrent* (melalui Goroutine):
      - Call API untuk Gas 1 (`device_id=12`)
      - Call API untuk Gas 2 (`device_id=13`)
      - Call API untuk Gas 3 (`device_id=14`)
   - Tampilkan hasil baca dan status pengiriman ke layar.
   - Tunggu selama 1 detik.
4. **Error Handling:** Log error dicetak jika Modbus gagal dibaca atau API target sedang *down*, aplikasi tetap lanjut ke iterasi berikutnya.

## 6. Kriteria Penerimaan (Acceptance Criteria)
- [ ] Aplikasi berhasil membaca data secara stabil dari target PLC.
- [ ] Konversi nilai Modbus sesuai dengan realitas di lapangan.
- [ ] Aplikasi berhasil melakukan HTTP GET request dengan parameter `device_id` dan `value` yang tepat untuk ketiga sensor.
- [ ] Proses *request* API tidak memperlambat/memblokir siklus *polling* 1 detik ke PLC (implementasi *non-blocking*).
- [ ] *Error handling* tertangani dengan baik apabila URL endpoint tidak dapat dijangkau.

---

## 7. Pengembangan Masa Depan (Roadmap)
- Pemisahan *Base URL* dan *Device ID* ke dalam file konfigurasi terpisah (`.env` atau `config.json`) agar tidak perlu me-rebuild kode saat URL atau ID berubah.
- Perubahan metode dari *multiple GET request* menjadi *single POST request* (mengirim data secara bulk/JSON) jika API endpoint mendukungnya di masa depan.
- Implementasi sistem Retry untuk HTTP request yang gagal.
